3.复现审计过程

漏洞处是在install.php下229~236行处

图中关于$_session检测是出题者自己加上去的，我们暂时忽略，把目标转向Typecho install.php存在的反序列化漏洞的审计。

我们可以看到代码231行调用Typecho_Cookie::get方法，读取__typecho_config的值。

我们跟进该get方法看一下，该方法位于Cookie.php的83～88行，从代码可以看到它会现在$_COOKIE中找__typecho_config，如果没有找到的话，就从post获取。

得到的__typecho_config会先用base64解密，然后反序列化。赋值后再将该值用delete方法进行处理

既然__typecho_config的值是由我们可以控制的，那现在可以看看哪里有可以让我们从这个可以控制的点调用的危险代码，并输出。

在反序列化之后，取出$config['adapter']和$config['prefix']实例化了一个Typecho_Db，跟进Typecho_Db

在上图的代码的120行，对传入的**$adapterName**进行了字符串拼接。

如果传入的**$adapterName**,是一个类，那么在将这个类进行字符串拼接的时候就会触发这个类的**__toString()**方法

call_user_func

(PHP 4, PHP 5, PHP 7)

call_user_func — 把第一个参数作为回调函数调用

说明

call_user_func ( callable $callback [, mixed $parameter [, mixed $... ]] ) : mixed

第一个参数 callback 是被调用的回调函数，其余参数是回调函数的参数。

array_map

(PHP 4 >= 4.0.6, PHP 5, PHP 7)

array_map — 为数组的每个元素应用回调函数

说明

array_map ( callable $callback , array $array1 [, array $... ] ) : array

array_map()**：返回数组，是为 array1 每个元素应用 callback函数之后的数组。 callback 函数形参的数量和传给 **array_map() 数组数量，两者必须一样。
